# Домашняя работа 4 — Go Core: map, методы, структуры и память

Домашняя работа закрепляет темы четвёртого занятия крупными логическими блоками. В каждом тематическом разделе находится 15 задач, после них — 3 объединяющие задачи. Условия полностью описаны в `docs/task.md` и TODO-комментариях.

## Разделы

| Раздел | Пакет | Количество задач | Темы |
|---|---|---:|---|
| Map | `internal/maps` | 15 | создание и изменение map, счётчики, выборка, индекс, группировка, склад значений и указателей |
| Методы | `internal/methods` | 15 | value/pointer receiver, ошибки, атомарность изменений, инварианты счёта и корзины |
| Структуры | `internal/structs` | 15 | структуры в map, глубокое копирование, embedding, promoted fields/methods, размеры и padding |
| Общие задачи | `internal/common` | 3 | обработка операций, индекс указателей и применение событий |

## Порядок выполнения

1. Прочитай общий раздел и таблицу нужного блока в `docs/task.md`.
2. Открой соответствующий файл в `internal/...`.
3. Реализуй функции вместо TODO, не меняя публичные сигнатуры.
4. Отформатируй код и запусти тесты только текущего блока.
5. После завершения всех разделов запусти полную проверку.

Тесты содержат минимум 10 сценариев на каждую функцию. В них проверяются обычные случаи, пустые входы, граничные значения, ошибки и отсутствие нежелательных изменений исходных данных.

## Команды

```bash
make compile
make fmt
make fmt-check
make test-maps
make test-methods
make test-structs
make test-common
make test-unit
make test-integration
make ci
```

Запуск демонстрационных программ:

```bash
make run-01_maps
make run-02_methods
make run-03_structs
make run-04_common
make run-all
```

## Анализ расположения полей структуры

После выполнения раздела со структурами можно отдельно проверить рекомендации по расположению полей:

```bash
go run golang.org/x/tools/go/analysis/passes/fieldalignment/cmd/fieldalignment@latest ./internal/structs/...
```

Анализатор показывает возможную экономию памяти, но его вывод не нужно применять механически: порядок полей может быть частью внешнего контракта или осознанного дизайна типа.

## Что нельзя изменять

Не изменяй тесты, Makefile, CI, сигнатуры функций, названия типов и sentinel-ошибки. Не добавляй готовые решения в README или `docs/task.md`.
